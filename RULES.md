# AppleInc 專案開發與架構規則 (Workspace Rules)

本文件基於初始的白板流程圖設計，定義後續架構與開發必須遵循的核心規則與系統運作流程。

## 1. 系統核心流程 (Core Workflow)
系統必須嚴格遵循以下五個階段的自動化管線 (Pipeline)：
1. **搜集 (Data Collection)**
2. **分析 (Data Analysis)**
3. **報告 (Reporting)**
4. **通知 (Notification)**
5. **排程 (Scheduling)**

## 2. 模組開發規範 (Module Specifications)

### 2.1 搜集模組 (Search & Collection)
- **資料來源**：必須包含「社群平台」、「網路新聞」與「公開訊息」。
- **實作方式**：需開發或整合適當的搜尋工具 (Search Tools) 與爬蟲/API，以自動化獲取 Apple Inc. 相關資訊。

### 2.2 分析模組 (LLM Analysis)
- **核心技術**：必須使用大型語言模型 (LLM) 作為分析引擎。
- **分析維度**：對搜集到的文本進行「情緒判定與分類」，必須至少輸出三種狀態：
  - `Positive` (正)
  - `Negative` (負)
  - `Neutral` / None (沒/中立)

### 2.3 報告模組 (Web Reporting)
- **呈現方式**：必須建構「網頁 (Web)」前端介面。
- **功能要求**：彙整 LLM 的分析結果，以視覺化或條列式在網頁上呈現報告。

### 2.4 通知模組 (Notification)
- **支援管道**：必須串接 `Line` 與 `Telegram` 的通訊 API。
- **觸發機制**：依據分析結果的嚴重性（例如重大負面新聞）或定時報告推播。

### 2.5 排程模組 (Task Scheduling)
- **執行頻率**：系統需具備定時執行能力，支援「每日 (Daily)」或自訂「多日」的週期性排程。
- **自動化**：確保 1 到 4 的步驟能完全自動化串接運行。

## 3. 開發與架構指導原則 (Architecture Guidelines)
- **微服務 / 模組化設計**：上述五個步驟應盡量解耦，以便獨立測試與抽換（例如更換 LLM 提供者或新增社群來源）。
- **錯誤處理 (Error Handling)**：在搜集與通知階段，需具備重試機制，避免單一來源失敗導致整個排程中斷。
- **語系要求**：程式碼註解可使用英文，但依照使用者全域設定，AI 的溝通說明與文件生成主要使用繁體中文。
- **事實查核**：AI 助理在答覆技術決策時，需遵循「證據優先」、「允許留白」、「來源標註」的內部原則。
